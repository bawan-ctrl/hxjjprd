# PRD-012:合伙人佣金管理（佣金锁 + 后台增减佣金）

**状态**：评审稿 · **日期**：2026-06-11 · **作者**：bawan

---

## 一、需求

平台每个用户都是合伙人。合伙人账户只做拉人和拿佣金，佣金不应随意流出变现。围绕合伙人佣金，后台提供两项**相互独立**的能力（展示与操作统一在用户管理内）：

1. **佣金锁**：在与合伙人约定活动等特定场景下，管理员对该合伙人开启佣金锁；开启后其**佣金余额禁止转出到现金余额**，佣金照常入账、只是停留在佣金余额，其他一切不变。
2. **后台增减佣金**：管理员可手动为合伙人**增加 / 扣除佣金余额**（如活动奖励发放、错误冲正），随时可用，不依赖佣金锁是否开启。

---

## 二、规则

**佣金锁：**

| 佣金锁 = 开 | 佣金锁 = 关 |
| --- | --- |
| 佣金余额 → 现金余额 划转 → 拦截，弹窗"账户受限，请联系客服" | 正常 |
| 佣金入账 → 正常（继续累计到佣金余额） | 正常 |
| 余额显示（佣金/现金）→ 正常 | 正常 |
| 现金余额提现 / 转入场馆 → 正常（不受本锁影响） | 正常 |
| 拉人/推广 → 正常 | 正常 |

**佣金 → 现金 划转与 AML：**

* 佣金 → 现金划转路径上**移除旧版 AML 锁**：划转金额不计入 AML 分段流水锁、不新增锁额，视为可直接提现，不要求再打流水。

**后台增减佣金：**

* 增加：佣金余额 += 金额；扣除：佣金余额 −= 金额，**不足则拒绝（不允许变负）**，提示余额不足。
* 金额用整型最小单位；必填备注；每次增减写**佣金流水 + 审计日志**，可对账。
* 账变在**数据库事务内 + 悲观锁** `SELECT … FOR UPDATE` 完成，并发安全；带 `partner_id` 隔离；提交幂等防重。

佣金锁不与 AML 锁、冻结余额等任何现有机制混淆，是独立布尔字段 `commission_lock`。

---

## 三、开发改动

**后台（admin）：**

* 用户管理 → 用户详情 → **账户信息**：展示**佣金余额**；同处放置「佣金锁」开关（+ 必填备注，操作写日志）与「增减佣金」操作（增加/扣除 + 金额 + 必填备注 + 二次确认，操作写审计日志）。

**接口改动：**

1. 佣金 → 现金 划转接口：请求前查 `user.commission_lock == true` → 返回拦截码（服务端权威）；**移除该划转路径上的旧版 AML 锁**，划转金额不写入 AML 分段流水锁。
2. 后台佣金调整接口：事务内 `SELECT … FOR UPDATE` 锁定佣金余额 → 校验（扣除不超额）→ 增减 → 写佣金流水 + 审计日志。

**前端（合伙人端）：** 佣金划转入口按钮置灰/隐藏；若仍发起 → 弹窗"账户受限，请联系客服" + 客服入口，不暴露原因。

其他接口、佣金入账、现金提现、转入场馆、AML —— **零改动**。

---

## 四、验收

**佣金锁：**

- [ ] 后台可开关佣金锁，必须填备注，操作有日志
- [ ] 佣金锁开 → 佣金→现金划转被拦截，弹窗联系客服
- [ ] 佣金锁开 → 佣金正常入账、余额正常显示、现金提现/转入场馆/拉人正常
- [ ] 佣金锁关 → 立即恢复划转
- [ ] 移除旧版 AML 锁后，佣金→现金划转金额不进入分段流水锁、可直接提现

**后台增减佣金：**

- [ ] 用户详情 → 账户信息中正确展示佣金余额，佣金锁开关与增减佣金入口均在此处
- [ ] 管理员可增加/扣除合伙人佣金，必填备注、二次确认
- [ ] 扣除超过当前佣金余额被拒绝、提示余额不足，余额不变负
- [ ] 每次增减生成佣金流水 + 审计日志，可对账
- [ ] 账变在事务内 + 悲观锁完成，并发不串账
