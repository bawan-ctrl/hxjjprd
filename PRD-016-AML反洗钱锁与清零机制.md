# AML 反洗钱锁与清零机制

**版本**:V2.0

**作者**:bawan(V1.0) / Ktufiur(V2.0 修订)

**V2.0 修订说明（对齐** [**PRD-017 分段式 AML 反洗钱流水锁**](https://hashrace.atlassian.net/wiki/spaces/Hashrace/pages/17334292)**）**

* **锁分段化**:AML 锁由「账户单一聚合锁」明确为「按每笔入金的分段锁」;清零作用于名下**所有未释放分段锁**(逐段解锁与按笔部分提现规则见 PRD-017)。
* **N 明确可配**:清零阈值 N 默认 5,后台可配(去除原「是否可配置待确认」)。
* **补可提现额度公式**:锁定额度 = Σ 未释放分段锁本金;可提现额度 = max(0, 总余额 − 锁定额度),含亏损保护。
* **举例口径对齐**:目标流水 = 该笔本金(1 倍),例 1 / 例 4 的「锁 2100」更正为「锁 150」。

---

## 一、差异对比

| **维度** | **现状** | **新方案** |
| --- | --- | --- |
| 清零依据 | aml  < 5 自动清零 | 总入金 − 总盈亏 ≤ N 自动清零(N 默认 5,后台可配) |
| 数据源 | 查三方场馆余额(需远程接口) | 本地入账与结算回调数据 |
| 受缓存滞后影响 | 是,可能误清 | 否 |
| 受接口超时影响 | 是,等价虚低 | 否 |
| 受未结算订单影响 | 是,可被藏钱钻空子 | 否,未结算自然不入账 |
| 真输光玩家体验 | 锁挂着,留存差 | 清零并设锚点,后续重新累计 |

---

## 二、解决的痛点

1. 缓存滞后偏低 → 误清锁
2. 多场馆并发查询数据不一致 → 计算结果不稳定
3. 三方接口超时、软失败返回 0 → 等价虚低 → 误清
4. 未结算订单藏钱攻击 → 绕过流水提现
5. 现行"锁本身 < 5 才释放" → 用户几千aml锁 不差这5 不是需求所需
6. 真输光的玩家旧锁不能合理释放 → 下次充值留存差 解决uiux

## 三、重要字段

| **字段** | **含义** |
| --- | --- |
| **总入金** | 锚点之后,所有让用户账户增加金额的事件累计:充值、红利、优惠券、礼金、佣金、返水、活动奖、调账入账等 |
| **总盈亏** | 锚点之后,**已成功结算并回调入账**的游戏盈亏累计。未结算/未回调一律不计入 |
| **时间锚点** | 上一次清零事件的时间戳;锚点之后开始重新累计 |
| **AML 锁(分段锁)** | 用户每完成**一笔入金**生成的一把独立锁,记录该笔的目标流水与已完成有效投注;锚点期内可并存多把。本机制的清零作用于名下**所有未释放分段锁**(逐段解锁见 PRD-017) |
| **目标流水** | 单把分段锁需完成的流水额度 = **该笔入金本金(1 倍)** |
| **剩余流水（跟剩余aml一个意思）** | 单把分段锁的 目标流水 − 已完成有效投注 |
| **锁定额度** | 所有**未释放**分段锁的本金之和,这部分钱暂不可提 |
| **可提现额度** | max(0, 总余额 − 锁定额度),用户当前真正可申请提现的金额 |
| **N(清零阈值)** | 默认 5,后台可配 |

## 五、业务方案

### 1. 核心规则

发生任一入金事件或结算回调事件时,系统立即:

* 更新当前锚点期的 总入金 / 总盈亏
* 计算 `总入金 − 总盈亏`
* ≤ N → 作废该用户名下所有未释放分段锁,并设新锚点;> N → 不动

### 2. 锁的形态:分段锁(V2.0 明确)

AML 锁按**每一笔入金**独立生成(分段锁),而非账户单一聚合锁:

* 每笔入金生成一把独立分段锁,目标流水 = 该笔本金(1 倍);锚点期内可并存多把。
* 正常情况下,有效投注按**先充先解(FIFO)**逐段填补、逐段解锁,解锁即可按笔部分提现(规则详见 PRD-017)。
* 本清零机制是兜底:净入金归零时,**一次性批量作废**名下所有未释放分段锁。

### 3. 可提现额度(V2.0 新增)

* **锁定额度** = 所有未释放分段锁的本金之和。
* **可提现额度 = max(0, 总余额 − 锁定额度)**。
* **亏损保护**:用户继续游戏导致总余额低于锁定额度时,可提现额度受保护归零,直到打满流水(逐段解锁)或触发本清零机制。

### 4. 时间锚点

清零瞬间记录时间锚点。锚点之后:

* 总入金、总盈亏从 0 重新累计
* 所有后续入金按各自规则产生新锁
* 锚点之前的锁标记为"已通过清零规则释放"

### 5. 延迟入金归属

入金按**到账时间**归属,不按事件产生时间。

例如:5/24 应得的佣金 5/25 09:00 到账 → 算 5/25 的入金。即使 5/24 23:00 已清零,这笔佣金仍计入新锚点期,产生新锁。

### 6. 评估时机（现有系统均已有功能）

| **事件** | **是否触发评估** |
| --- | --- |
| 充值 / 红利 / 优惠券 / 礼金到账 | 是 |
| 佣金 / 返水到账(含延迟) | 是 |
| 活动奖 / 客服入账型调账 | 是 |
| 游戏订单**结算回调成功** | 是 |
| 游戏下注(订单刚产生未结算) | 否 |
| 登录、查余额、看页面 | 否 |

### 7. 异常处理

| **情况** | **处理** |
| --- | --- |
| 订单未结算 | 不入总盈亏 |
| 回调延迟到达 | 暂不更新总盈亏,锁保守保留;到达后再评估 |
| 回调丢失 | 锁保守保留;依靠**每日对账**找回补入账 |
| 重复回调 | 幂等,同一订单只入账一次 |
| 客服调账 | 入账型 → 算入金;扣款型 → 算盈亏 |
| 撤销 / 退款 | 反向冲销原入金或原盈亏 |

### 8. 清零后冷静期(可由运营开关)

* 锚点后首次提现走自动复核 或者提醒（待商议）

## 六、举例说明

### 例 1:正常输光

充 100 + 红利 50,总入金 150,锁 150(每笔本金 1 倍)。 陆续输 150 → 总盈亏 150。 **150 − 150 = 0 ≤ 5 → 清零**,锚点 T1。

### 例 2:输光后第二天佣金到账

5/24 23:00 输光,清零,锚点 T1。 5/25 09:00 佣金 50 到账(1 倍流水)。 归入新锚点期 → 总入金 50,新锁 50。 不玩就挂着,再输光走新一轮清零。

### 例 3:中途盈利

充 100,锁 100。赢 50 → 总盈亏 = -50。 **100 − (-50) = 150 > 5 → 锁保留**(合理,他还有钱)。

### 例 4:藏钱攻击(失败)

充 100 + 红利 50,总入金 150,锁 150。 转 145 到棋牌,下大额未结算注 140。 此时总盈亏 = 0(注未结算)。 **150 − 0 = 150 ≫ 5 → 清不掉**,提现被拒。

### 例 5:多笔入金叠加输光

充 1000 → 入金 1000,锁 1000。 输 600 → 盈亏 600,差 400,不清。 再充 500 → 入金 1500,差 900,不清。 再输 900 → 盈亏 1500,差 0 → **清零,两笔锁同批释放**。

### 例 6:回调延迟

实际输 1000,但有 50 的回调还没到。 当前总盈亏 = 950,差 50 > 5,**锁保留**。 50 回调到达后 → 总盈亏 1000,差 0 → 清零。

## 七、重要点验收

| **#** | **用例** | **期望** |
| --- | --- | --- |
| 1 | 真输光,所有回调成功 | 清零,锚点更新 |
| 2 | 输光后第二天佣金到账 | 计入新锚点期,产生新锁 |
| 3 | 用户中途盈利 | 锁保留 |
| 4 | 转账场馆 + 大额未结算注 | 公式不变,清不掉 |
| 5 | 三方接口超时 / 失败 | 不影响判定 |
| 6 | 回调延迟到达 | 锁保守保留;到达后正确评估 |
| 7 | 同一订单回调两次 | 幂等,只入账一次 |
| 8 | 清零后再充值 | 新锚点期独立产生新锁 |
| 9 | 客服调账 +100 入账 | 计入总入金,产生新锁 |
| 10 | 充两笔仅打满一笔(未触发清零) | 对应分段锁解锁,可按笔部分提现(详见 PRD-017) |
| 11 | 用户输光但有未到的回调 | 锁继续保留,等回调到再评估！！ |
