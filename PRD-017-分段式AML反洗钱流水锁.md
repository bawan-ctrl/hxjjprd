# 分段式 AML 反洗钱流水锁

**版本**:V1.0  
**作者**:ktufiur  
**关联**:本 PRD 的「真输光自动清零」机制完全沿用 [PRD-016 AML 反洗钱锁与清零机制](https://hashrace.atlassian.net/wiki/spaces/Hashrace/pages/5308529) V2.0

---

## 一、产品概述

**业务背景**:为满足反洗钱(AML)合规要求,防止平台被当作资金过桥或"洗白"通道,用户充值的资金必须经过真实游戏消耗后才能提现。

**当前痛点**:现有流水锁是"一个账户一把大锁",所有充值揉在一起算流水。用户无法在打满一部分流水后先提走那部分钱——要么全锁要么全开,提现体验差、留存差。

**本期目标**:

* **按笔独立追踪**——每一笔充值生成一把独立的流水锁(分段锁),互不干扰。
* **按笔部分提现**——只要某一笔充值的流水打满,这笔钱就能立刻提现,不必等所有充值都达标。
* **渠道差异化要求**——不同游戏渠道对流水的贡献不同(老虎机宽松、真人视讯严格),后台可配。
* **真输光自动解套**——用户真把钱输光时,历史欠下的流水自动一笔勾销,不影响下次充值体验(沿用 PRD-016)。

**用户故事**:

> 作为玩家,我充了两笔钱,只把第一笔的流水打满了。我希望能先把第一笔对应的钱提出来,而不是被第二笔的流水卡住整个账户。

## 二、名词解释

| 名词 | 含义 |
| --- | --- |
| **分段锁** | 用户每完成一笔充值,系统生成的一把独立流水任务,记录「目标流水」和「当前进度」。 |
| **目标流水** | 该笔充值需要完成的流水额度。**= 该笔充值本金(1 倍)**。 |
| **有效流水** | 实际投注额经渠道权重折算后、真正用于解锁的流水数值。 |
| **渠道权重** | 某渠道「要打几倍投注额才算满 1 倍有效流水」。DB 3 倍、体育 3 倍、HR 斗地主 1 倍,后台可配。 |
| **锁定额度** | 所有「未解锁」分段锁的本金之和,这部分钱暂不可提。 |
| **可提现额度** | 用户当前真正可以申请提现的金额。 |
| **真输光自动清零** | 用户净充值接近归零时,系统自动作废其名下所有未解锁分段锁(判定与机制见 PRD-016)。 |

## 三、核心玩法规则

### 3.1 分段锁的产生

用户每成功完成一笔充值(法币或数字资产入账),系统**按充值时间顺序**生成一把独立的分段锁。

> **示例**:先充 100 CNY → 生成【锁 A:目标 100,进度 0】。随后再充 500 CNY → 生成【锁 B:目标 500,进度 0】。两把锁各自独立计算。

### 3.2 渠道权重与流水折算

不同游戏渠道对流水的贡献度不同,通过后台配置「渠道权重」实现差异化。

**折算公式:有效流水 = 实际投注额 ÷ 渠道权重**

| 渠道(Channel) | 渠道权重(投注额倍数) | 案例:实际下注 300 CNY |
| --- | --- | --- |
| DB | 3 倍投注额 | 300 ÷ 3 = 产生 100 CNY 有效流水 |
| 体育投注 Sports | 3 倍投注额 | 300 ÷ 3 = 产生 100 CNY 有效流水 |
| HR 斗地主  | 1 倍投注额 | 300 ÷ 1 = 产生 300 CNY 有效流水 |

> **理解**:渠道权重越大,要求打的投注额越多。例如体育投注 3 倍,意味着要下注 3 倍金额才折算成 1 倍有效流水。

### 3.3 解锁顺序:先充先解(FIFO)

有效流水产生后,**永远优先填补"创建最早、且尚未解锁"的那把分段锁**:

* **填满即解锁**:某把锁的累计有效流水 ≥ 目标流水时,该锁状态变为「已解锁」,对应这笔充值的钱即可提现。
* **超出顺延**:单次产生的有效流水若超过当前锁的剩余需求,多出的部分自动流转到下一把未解锁的锁,不浪费。

### 3.4 真输光自动清零(沿用 PRD-016)

为避免"真输光的玩家"还背着历史流水欠账、影响下次充值体验,设置兜底:

* **判定依据**:锚点期内 **总入金 − 总盈亏 ≤ N**(N 默认 5 CNY,后台可配)。
* **执行动作**:作废该用户名下**所有未解锁分段锁**,并设新时间锚点;之后重新累计。
* **数据来源**:完全基于本地入账与结算回调数据,**不查询三方场馆余额**,因此不受缓存滞后、接口超时、未结算订单藏钱攻击影响。

完整判定规则、字段定义、异常处理与举例,以 **PRD-016 V2.0** 为准。本 PRD 只声明「分段锁的批量作废」由该机制驱动。

### 3.5 风控规则

| 场景 | 处理 |
| --- | --- |
| **注单取消 / 退款** | 赛事取消或故障退回本金的注单不计有效流水。若该注单此前已结算并解开过某把锁,需**回滚**:恢复对应锁的锁定状态与进度。 |

## 四、提现规则(部分提现 + 亏损保护)

用户的钱按"锁定 / 可提"两部分管理:

* **总现金余额** = 钱包实际拥有的 CNY 总数。
* **锁定额度** = 所有未解锁分段锁的本金之和。
* **可提现额度 = max(0, 总余额 − 锁定额度)**。

### 推演 1:正常部分提现

充 100(锁 A)+ 200(锁 B),总余额 300,锁定 300,可提 0。  
玩游戏赢钱后总余额变 400,并打满了锁 A → 仅剩锁 B,锁定 = 200。  
**可提现 = 400 − 200 = 200**,用户可立即提现 200 CNY。

### 推演 2:输钱时的亏损保护

接上例,用户没提现,继续玩输到总余额只剩 150 CNY,锁 B 仍未解锁(锁定仍 200)。  
**可提现 = max(0, 150 − 200) = 0**,提现额度受保护归零——必须继续打满流水,或触发真输光自动清零后才能动这部分。

## 五、用户端体验

### 5.1 提现AML锁列表

* 顶部标题、AML 政策说明+「更多详情」、按笔充值卡片(您的充值+日期 / 金额 / 总押注 当前÷目标 + 百分比 /  进度条,已解锁绿色高亮)、底部关闭按钮。

### 5.2 锁列表触发入口

* 点击「锁定额度」字段,或提现被拦截时,弹出该锁列表

### 5.3 交互提示

* 输入金额 ≤ 可提现额度 → 正常进入提现流程。
* 输入金额 > 可提现额度但 ≤ 总余额 → 提现按钮置灰或点击飘红,提示如下。

"您的部分余额因未满足充值流水要求被锁定,当前最多可提现 {可提现额度} CNY。请查看下方流水进度详情。"

## 六、管理后台

### 6.1 玩家分段锁列表

运营 / 管理员可按在玩家详情中查看该玩家的全部分段锁:

* 展示字段:关联充值订单号、目标流水(本金)、当前有效流水进度、剩余流水、渠道分布、状态(未解锁 / 已解锁 / 已作废)、创建时间、所属清零锚点。

### 6.2 管理员手动作废单个分段锁

管理员可对某一把「未解锁」分段锁单独执行**作废**(区别于系统自动的批量清零):

* **效果**:该笔本金从「锁定额度」移出,对应提现权立即释放;页面可提现额度实时刷新。
* **管控要求**(资金敏感操作):需管理员相应权限(普通运营无权);**强制填写作废原因**并二次确认;全程记录审计日志(操作人、原因、时间、作废前后额度快照);建议走「申请 — 复核」双人流程(maker-checker)。
* **后续流水处理**:已作废的锁不再参与先充先解的填补,后续有效流水自动顺延到下一把未解锁的锁。

## 七、验收用例

| # | 用例 | 期望 |
| --- | --- | --- |
| 1 | 充两笔,打满第一笔流水 | 第一笔锁解锁,对应金额可提现;第二笔仍锁定 |
| 2 | DB下注 300 | 产生 100 有效流水 |
| 3 | 斗地主下注 300 | 产生 300 有效流水 |
| 4 | 单笔有效流水超过当前锁剩余需求 | 超出部分顺延填补下一把锁 |
| 5 | 已打满锁后继续输钱致余额低于锁定额度 | 可提现额度受保护归零 |
| 6 | 真输光(总入金−总盈亏≤N) | 名下所有未解锁分段锁批量作废,设新锚点 |
| 7 | 已解锁注单被退款 | 回滚:恢复对应锁的锁定与进度 |
| 8 | 提现金额 > 可提现额度 | 拦截并提示最高可提金额 |
| 9 | 管理员作废某把未解锁锁 | 该笔本金释放为可提,留审计日志 |
| 10 | 后台查看某玩家分段锁列表 | 正确展示各锁状态与进度,租户隔离 |

## 八、边界与待定

* 提现进入审核时,申请金额需即时从可提现额度中冻结,防止时间差导致重复提现或二次下注(实现细节留待 spec)。
* 游戏结算频率高,流水进度与额度的重算建议异步处理,结算后实时刷新前端(实现细节留待 spec)。
* 渠道权重的默认值与可配粒度(按渠道 / 按游戏)在后台配置 spec 中细化。
* 「真输光自动清零」的阈值 N、冷静期开关等以 PRD-016 V2.0 为准。
